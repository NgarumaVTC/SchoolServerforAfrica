* Overview

This document describes the steps to restore the NixOS school server from the
ZFS-based backup. The procedure is designed to be executable by the remote admin
in Germany via WireGuard VPN where possible, and on-site otherwise.

** Backup Details
- Backup target: 2× Transcend StoreJet 25M3C USB-C drives (weekly rotation, on-site)
- Off-site copy: weekly ZFS send to admin machine in Germany (via WireGuard)
- Method: =zfs send | zfs receive= (incremental snapshots, compressed)
- NixOS system config: Git repository (GitHub: NgarumaVTC/gitNixos)

** Recovery Scenarios

| Scenario                       | Recovery Path                                  |
|--------------------------------+-------------------------------------------------|
| Misconfiguration / bad update  | =nixos-rebuild switch --rollback=              |
| ZFS dataset corruption         | =zfs rollback= to last snapshot                |
| Single drive failure (mirror)  | =zpool replace= — no downtime                  |
| Full disk failure + OS lost    | This document (Phases 1–4)                     |
| Total hardware loss            | Replace hardware, follow Phases 1–4 from USB   |

* Phase 1: Environment Setup

1. Boot the server using the *NixOS Minimal Live USB* (keep one labelled and stored
   next to the server).
2. Set keyboard layout:
   #+BEGIN_SRC bash
   loadkeys de
   #+END_SRC
3. Verify network connectivity to backup source:
   #+BEGIN_SRC bash
   # If recovering from on-site USB drive:
   lsblk   # identify the Transcend backup drive

   # If recovering from off-site (Germany) via VPN — WireGuard must be
   # configured on the live USB or another network path must be available.
   ping -c 3 172.20.0.254
   #+END_SRC

* Phase 2: Partitioning & Bootloader

** 1. Restore Partition Table (GPT)
#+BEGIN_SRC bash
# Restore GPT from backup (replace source path as appropriate)
cat /mnt/backup/ngarumavtc1_gpt.bak | sgdisk --load-backup=/dev/stdin /dev/sda
sgdisk --load-backup=/tmp/ngarumavtc1_gpt.bak /dev/sda
partprobe /dev/sda
# Repeat for /dev/sdb and /dev/sdc (3-way mirror)
#+END_SRC

** 2. Restore EFI Partition
#+BEGIN_SRC bash
cat /mnt/backup/ngarumavtc1_efi.img.zst | zstd -d | dd of=/dev/sda1 status=progress
#+END_SRC

* Phase 3: ZFS Pool Recovery

** 1. Create Fresh ZFS Pool (if drives are replaced)
#+BEGIN_SRC bash
# 3-way mirror on the three datacenter SSDs
zpool create -f \
  -o ashift=12 \
  -O compression=zstd \
  -O atime=off \
  -O xattr=sa \
  -O mountpoint=none \
  -R /mnt \
  rpool mirror /dev/sda2 /dev/sdb2 /dev/sdc2

# 2-drive mirror for student data
zpool create -f \
  -o ashift=12 \
  -O compression=zstd \
  -O atime=off \
  -O xattr=sa \
  -O mountpoint=none \
  tank mirror /dev/sdd /dev/sde
#+END_SRC

** 2. Restore Data via ZFS Receive
#+BEGIN_SRC bash
# From on-site USB backup drive (mounted at /mnt/backup):
cat /mnt/backup/rpool_full.zst | zstd -d | zfs receive -F rpool
cat /mnt/backup/tank_full.zst  | zstd -d | zfs receive -F tank

# From Germany via SSH over WireGuard:
ssh ramge@172.20.0.254 "cat ~/backup/rpool_full.zst" | zstd -d | zfs receive -F rpool
#+END_SRC

* Phase 4: NixOS System Reinstall from Git

If the NixOS system dataset is lost or corrupted, reinstall from the Git config:

#+BEGIN_SRC bash
# 1. Mount target filesystems
mount -t zfs rpool/nixos/root /mnt
mkdir -p /mnt/{boot,nix,var}
mount -t zfs rpool/nixos/nix /mnt/nix
mount -t zfs rpool/nixos/var /mnt/var
mount /dev/sda1 /mnt/boot

# 2. Clone the NixOS config from GitHub
nix-shell -p git
git clone https://github.com/NgarumaVTC/gitNixos /mnt/etc/nixos

# 3. Rebuild and install
nixos-install --root /mnt
#+END_SRC

* Phase 5: Finalization

#+BEGIN_SRC bash
# Export pools cleanly before reboot
zpool export rpool
zpool export tank

reboot
# Remove USB stick
#+END_SRC

After reboot, verify:
- [ ] NixOS boots cleanly to login prompt
- [ ] All ZFS pools import automatically (=zpool status=)
- [ ] All containers start (=nixos-container list=)
- [ ] WireGuard VPN tunnel re-establishes
- [ ] IPMI accessible from Germany
- [ ] PXE boot of one thin client successful
